The Core Idea

Normal flow vs. the shortcut

Full OAuth UI
✦ Build a "Connect Google Calendar" button
✦ Handle the redirect callback route
✦ Exchange the code server-side
✦ Store & refresh tokens over time
✦ Handle disconnect / re-auth flows
~3–4 hrs · code you use once
The Shortcut
✦ Click through Google's own OAuth Playground
✦ Copy the refresh token
✦ Paste it into Supabase secrets
✦ ~~No button to build~~
✦ ~~No callback route~~
~15 min · works for the life of the app

Refresh tokens don't expire unless you revoke them or change your Google password. Single-user app = you're always the user. There's nobody else to log in.

Step 1 of 4 · ~10 min

Google Cloud Console

1
Create a project
console.cloud.google.com → New Project → name it anything (e.g. "GTD Manager")
2
Enable the Calendar API
APIs & Services → Library → search "Google Calendar API" → Enable
3
Create OAuth credentials
Credentials → Create Credentials → OAuth 2.0 Client ID → type: Web Application
Add https://developers.google.com/oauthplayground as an Authorized Redirect URI
4
Copy your credentials
You'll get a Client ID and a Client Secret. Keep them handy for the next step.
Step 2 of 4 · ~5 min

OAuth Playground → grab the refresh token

1
Open OAuth Playground
developers.google.com/oauthplayground
2
Use your own credentials
Click the ⚙️ gear (top-right) → check "Use your own OAuth credentials"
Paste in your Client ID and Client Secret from Step 1
3
Select scope & authorize
In the scope box, paste:
https://www.googleapis.com/auth/calendar.readonly
→ Authorize APIs → sign in with your Google account → Allow
4
Exchange code for tokens
Click "Exchange authorization code for tokens"
You'll see a JSON response. Copy the refresh_token value — that's the prize.
Step 3 of 4 · ~2 min

Store secrets in Supabase

Supabase Dashboard → Edge Functions → Secrets (or use the CLI). Add three secrets:

GOOGLE_CLIENT_ID = 1234567890-abc...apps.googleusercontent.com GOOGLE_CLIENT_SECRET = GOCSPX-your-client-secret GOOGLE_REFRESH_TOKEN = 1//04abc...long-refresh-token-string
Client ID

Safe to put in Vite frontend too as VITE_GOOGLE_CLIENT_ID — it's public by design

Client Secret

Never in the frontend bundle. Edge Function only.

Refresh Token

The one you just copied. Never expires. Never leaves the server.

Step 4 of 4 · ~1–2 hrs of code

The Supabase Edge Function

This is the only real code to write. It runs on every page load of Daily.

// supabase/functions/google-calendar/index.ts // Called by the frontend: GET /functions/v1/google-calendar // 1. Read secrets from env clientId = Deno.env.get('GOOGLE_CLIENT_ID') clientSecret = Deno.env.get('GOOGLE_CLIENT_SECRET') refreshToken = Deno.env.get('GOOGLE_REFRESH_TOKEN') // 2. Exchange refresh token → fresh access token (auto, every call) accessToken = await fetchNewAccessToken(clientId, clientSecret, refreshToken) // 3. Fetch today's calendar events events = await fetch(`calendar/v3/calendars/primary/events ?timeMin=today-00:00&timeMax=today-23:59 &singleEvents=true&orderBy=startTime`, { Authorization: accessToken }) // 4. Return simplified shape to frontend return events.map(e => ({ id, title, start, end, location, allDay }))

No token storage in a database. No refresh-token rotation logic. Just: read secret → get access token → get events → return.
